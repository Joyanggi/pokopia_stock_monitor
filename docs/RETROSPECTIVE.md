# Pokopia Stock Monitor 프로젝트 회고

이 문서는 코드와 Git 커밋에서 확인되는 사실, 그리고 프로젝트의 개인 사용 목적을 달성했다는 현재 상태를 바탕으로 작성한 포트폴리오용 기록입니다. 확인되지 않은 구매 결과나 성과 수치는 넣지 않았습니다.

## 1. 한 문장 소개

빠르게 올라왔다가 종료되는 포켓몬 포코피아 판매 정보를 놓치지 않기 위해, 루리웹 핫딜 게시판을 30초마다 확인하고 유효한 새 글만 데스크톱으로 알려 주는 Python 자동화 도구를 만들었습니다.

## 2. 문제 상황

포코피아 관련 판매 정보는 한 판매처에 고정되어 있지 않고 커뮤니티 게시판에 간헐적으로 공유됐습니다. 직접 확인하는 방식에는 세 가지 문제가 있었습니다.

1. 새 글이 언제 올라올지 몰라 게시판을 계속 새로고침해야 했습니다.
2. 검색 결과에는 이미 종료되거나 품절된 판매, 다운로드판처럼 찾는 대상이 아닌 글도 섞였습니다.
3. 같은 글을 반복해서 발견하면 알림의 신뢰도가 떨어집니다.

핵심 과제는 거대한 재고 관리 서비스를 만드는 것이 아니라, **개인적인 구매 탐색 과정에서 반복되는 확인 작업을 작고 설명 가능한 자동화로 바꾸는 것**이었습니다.

## 3. 해결 과정

### 3.1 여러 판매처 대신 정보가 모이는 지점을 선택

판매처별 페이지를 각각 크롤링하면 사이트마다 구조와 예외 처리가 달라집니다. 이 프로젝트는 여러 판매 정보가 모이는 루리웹 핫딜/예판 게시판을 하나의 신호원으로 선택했습니다.

- 장점: 구현 범위가 작고 새로운 판매처의 글도 같은 흐름에서 감지할 수 있음
- 대가: 커뮤니티 글이 실제 재고를 보장하지 않으며, 글이 올라오기 전에는 감지할 수 없음

### 3.2 짧은 주기의 폴링으로 발견 지연 축소

공식 API나 웹훅이 없는 게시판이므로 `requests`로 목록 페이지를 30초마다 요청하고 `BeautifulSoup`으로 게시글의 제목·링크·ID를 추출했습니다.

30초는 즉시성, 개인 PC의 부담, 대상 사이트 요청량 사이에서 택한 단순한 기본값입니다. 상수로 분리해 상황에 따라 쉽게 조정할 수 있게 했습니다.

### 3.3 포함 규칙과 제외 규칙을 분리

상품명 표기를 놓치지 않도록 `포코피아`, `포켓몬 포코피아`, `Pokopia`를 포함 키워드로 두었습니다. 동시에 `종료`, `품절`, `마감`, `다운로드`, `DL판`, 잠금 표시를 제외해 원하는 패키지 판매 신호에 집중했습니다.

머신러닝 분류보다 단순한 문자열 규칙을 선택한 이유는 데이터가 적은 개인 도구에서 판단 근거가 명확하고, 오탐이 생겼을 때 즉시 규칙을 수정할 수 있기 때문입니다.

### 3.4 게시글 ID를 로컬에 남겨 중복 방지

매 주기마다 같은 목록을 읽더라도 한 번 확인한 게시글은 다시 알리지 않도록 ID를 `seen_ids.json`에 저장했습니다. 별도 DB 없이도 프로세스를 재시작한 뒤 상태를 이어 갈 수 있고, 파일이 끝없이 커지지 않도록 최근 500개로 제한했습니다.

### 3.5 발견과 행동 사이의 시간을 줄이는 OS 알림

터미널 로그만으로는 사용자가 화면을 보고 있지 않을 때 놓칠 수 있습니다. Windows에서는 `plyer`와 시스템 소리, macOS에서는 알림 센터와 효과음을 사용해 새 글을 발견한 즉시 주의를 환기했습니다.

## 7. 이력서에 사용할 수 있는 문장

### 한 줄 버전

루리웹 핫딜 게시판의 포코피아 판매 글을 30초 주기로 탐지하고 중복·종료 글을 제거해 데스크톱 알림으로 전달하는 Python 모니터링 도구를 개발했습니다.

### 불릿 버전

- 수동 새로고침이 필요했던 한정 상품 탐색 과정을 `Requests`와 `BeautifulSoup` 기반의 30초 주기 모니터링으로 자동화
- 포함·제외 키워드를 분리해 상품명 변형은 포착하고 종료·품절·다운로드판은 걸러내는 설명 가능한 필터 설계
- 게시글 ID를 로컬 JSON에 최대 500개까지 보존해 재시작 후에도 중복 알림을 방지하고, Windows/macOS 네이티브 알림으로 발견 즉시 전달
- 공식 재고 API가 없는 제약에서 커뮤니티 게시판을 대체 신호로 선택하고, 개인 목적 달성 후 유지 비용을 고려해 프로젝트를 명확히 종료

### STAR 형식

- **Situation**: 포코피아 관련 판매 글이 간헐적으로 올라와 게시판을 계속 직접 확인해야 했습니다.
- **Task**: 관련 새 글을 빠르게 발견하면서 종료 글과 중복 알림을 줄이는 가벼운 개인 도구가 필요했습니다.
- **Action**: 게시판 폴링, 제목 기반 포함·제외 필터, 게시글 ID 영속화, 운영체제별 데스크톱 알림을 단일 Python 프로그램으로 구현했습니다.
- **Result**: 반복 확인을 자동화해 개인 사용 목적을 달성했고, 운영 가치가 사라진 시점에 완료 상태로 전환해 설계 판단과 한계를 문서로 남겼습니다.

## 8. 면접에서 설명할 포인트

### “왜 판매처를 직접 크롤링하지 않았나요?”

개인 문제를 빠르게 해결하는 것이 목표였기 때문에 판매 정보가 모이는 커뮤니티를 신호원으로 선택했습니다. 여러 판매처를 각각 연동하는 것보다 구현과 유지 범위를 줄이는 대신, 실제 재고를 보장할 수 없다는 한계를 README에 명시했습니다.

### “왜 DB가 아니라 JSON인가요?”

사용자는 한 명이고 프로세스도 하나였습니다. 필요한 상태는 이미 본 게시글 ID뿐이어서 DB의 설치·운영 비용보다 로컬 JSON이 문제 크기에 더 적합했습니다.

### “왜 키워드 규칙인가요?”

분류 데이터가 없는 작은 도구에서는 복잡한 모델보다 규칙이 빠르고 설명 가능합니다. 포함어와 제외어를 분리해 오탐 원인을 바로 추적하고 수정할 수 있게 했습니다.

### “다시 만든다면 무엇을 개선하겠나요?”

실사용 범위를 넓힌다면 다음 순서로 개선할 수 있습니다.

1. 요청 실패 시 지수 백오프와 구조화된 로그 추가
2. HTML 선택자 변경을 감지하는 파서 테스트와 샘플 fixture 추가
3. 키워드와 주기를 환경 변수나 설정 파일로 분리
4. 텔레그램·푸시 채널을 추가해 PC가 꺼져 있어도 알림 수신
5. 가능하다면 판매처의 공식 API나 알림 채널로 신호원 교체

현재 프로젝트에는 이 확장을 넣지 않았습니다. 개인 목적을 이미 달성한 뒤 기능을 계속 늘리는 것보다, 종료 상태와 판단 근거를 명확히 남기는 편이 더 적절하다고 판단했습니다.
